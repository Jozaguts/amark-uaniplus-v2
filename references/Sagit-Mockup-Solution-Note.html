Nota técnica para Sagit · Mockup en vivo vs. producción

El problema: el Mockup Generator de Printful tarda ~2 min. Eso NO sirve para una vista previa en vivo. La solución: separa la vista previa (instantánea, en el navegador) de la producción (Printful, en segundo plano, después del pago).

1. El diagnóstico (por qué está atascado)

El Mockup Generator de Printful es asíncrono y basado en tareas: tú envías un trabajo → Printful lo renderiza en sus servidores → tú consultas (poll) el resultado. Está diseñado para generar catálogo en segundo plano, NO para una vista previa en tiempo real mientras el cliente mira la pantalla.

Conclusión clave: nunca vas a hacerlo lo suficientemente rápido. No intentes acelerar a Printful. En vez de eso, deja de usar Printful para la vista previa en vivo. Usa Printful SOLO al final, para el archivo de producción.

2. La arquitectura correcta — separa en dos

┌─────────────────────────────────────────────────────────────┐
│  VISTA PREVIA (instantánea, en el navegador, SIN API)         │
│  El cliente diseña y ve el resultado al instante en el canvas │
│  → Fabric.js: foto del modelo de fondo + diseño encima        │
└─────────────────────────────────────────────────────────────┘
                          │  (el cliente paga)
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  PRODUCCIÓN (lenta, en segundo plano, DESPUÉS del checkout)   │
│  Job en cola (Horizon) → Printful Mockup + Order API          │
│  El cliente YA pagó y recibió confirmación. No espera nada.   │
└─────────────────────────────────────────────────────────────┘

Los ~2 minutos de Printful ahora ocurren en segundo plano, después del pago, donde no molestan a nadie.

3. La vista previa en vivo (la parte que lo desatasca HOY)

Ya estás usando Fabric.js para que el cliente diseñe. Esa misma canvas es la vista previa. No necesitas a Printful para esto.

Opción A — Overlay con Fabric.js (recomendada, simple, ya casi la tienes)

La idea: la foto del modelo va como fondo, y el diseño del cliente se renderiza encima, dentro del área de impresión, con un poco de mezcla para que parezca tela.

// 1. Foto del modelo como fondo de la canvas (NO editable)
fabric.Image.fromURL(modelPhotoUrl, (img) => {
  img.set({ selectable: false, evented: false });
  canvas.backgroundImage = img;
  canvas.renderAll();
});

// 2. El diseño del cliente (texto + arte) va en una zona definida
//    = el "print area" mapeado sobre la foto del modelo.
const printArea = { left: 220, top: 180, width: 260, height: 340 }; // por producto/variante

// 3. Clip: el diseño NO se sale del área de impresión
designGroup.clipPath = new fabric.Rect({
  left: printArea.left, top: printArea.top,
  width: printArea.width, height: printArea.height,
  absolutePositioned: true
});

// 4. Que parezca impreso sobre tela (no un sticker plano)
designGroup.set({
  opacity: 0.92,
  globalCompositeOperation: 'multiply' // se funde con sombras/pliegues de la tela
});
canvas.add(designGroup);
canvas.renderAll(); // INSTANTÁNEO — sin llamada a API

Por qué funciona: cada cambio del cliente (escribir, mover, color) se ve al instante porque todo pasa en el navegador. Cero latencia, cero costo de API.

Opción B — Si quieres que se vea más realista (después, no ahora)

  • Una capa de displacement map (mapa de desplazamiento) de la tela para que el diseño siga los pliegues. Más realista, más trabajo.
  • O una librería tipo smart-object en canvas.
  • No lo necesitas para lanzar. El overlay de la Opción A es suficiente para "escribe tu texto → míralo en la camiseta".

El punto honesto

La vista previa hecha por nosotros no será idéntica a la foto final de Printful. Está bien y es lo normal: - Etiquétala como "Vista previa" (preview). - Usa tus fotos reales de modelos talla grande (Rawpixel / Placeit) como las imágenes "hero" del producto. - El overlay en vivo es la capa interactiva, no la foto de estudio. Nadie espera que una vista previa en vivo sea una foto profesional — espera que sea instantánea.

4. La producción (la parte lenta, oculta)

Solo DESPUÉS del checkout:

  1. Aplana el diseño del cliente a un archivo de impresión PNG con el DPI y dimensiones exactas que pide Printful para esa variante (no resolución de pantalla — si no, sale borroso).
  2. Lanza un job en cola (Horizon): - (opcional) genera el mockup oficial de Printful para el correo/historial — async, sin bloquear. - crea la orden en Printful usando el Variant ID (NUNCA el Product ID) + el archivo + la posición.
  3. Guarda printful_order_id y el estado. Maneja los webhooks de Printful para fulfillment/envío → manda el tracking al cliente.

El cliente ya pagó y vio su confirmación. Los 2 minutos ya no le importan a nadie.

5. Estructúralo para que iPlusLove sea un FORK limpio

iPlusLove = uandiplus menos CJDropshipping, más el filtro de talla grande. Estás borrando un módulo, no añadiendo uno. Para que el copy-paste sea trivial, haz esto AHORA en uandiplus:

  • El proveedor de fulfillment debe ser configuración, no código fijo. FULFILLMENT_DRIVER = printful | cjdropshipping. En iPlusLove simplemente apagas CJ.
  • La fuente del catálogo debe ser configuración. Un filtro/flag, no IDs escritos a mano.
  • El overlay de mockup (foto del modelo + print area) debe leerse de config/DB por producto, no hardcodeado. Así iPlusLove solo cambia las fotos de modelos (a talla grande) y el área de impresión.
  • El filtro size_inclusive (que ya está en el blueprint de iPlusLove) es básicamente el único añadido nuevo: muestra un producto solo si Printful lo ofrece en tallas grandes (calcula el flag desde los datos en vivo, no a mano).

Si lo construyes así, iPlusLove es un fork con dos o tres settings cambiados, no una reconstrucción.

6. Resumen en una línea (para Sagit)

Vista previa = Fabric.js en el navegador, instantánea, sin API. Printful = solo para producción, en un job en cola, después del pago. Y deja el fulfillment + catálogo + overlay como configuración para que iPlusLove sea un fork limpio.

7. English summary (quick reference)

  • Don't use Printful's Mockup Generator for the live preview — it's async (~2 min), built for back-office catalog generation, not real-time. You'll never make it fast enough.
  • Live preview = render it yourself client-side with Fabric.js: model photo as backgroundImage, customer design as a clipped group inside the print area, multiply blend + slight opacity so it reads as printed-on-fabric. Instant, no API call. You're already on Fabric — the design canvas is the preview.
  • Production = after checkout only: flatten to a print-ready PNG at Printful's required DPI/dimensions per variant, then a queued Horizon job creates the Printful order by Variant ID (never Product ID). Customer already paid; the 2 minutes happens in the background.
  • Label the live preview "Preview"; use real plus-size model photos (Rawpixel/Placeit) as the polished hero images. A live preview is expected to be instant, not a studio shot.
  • Make it fork cleanly: fulfillment driver, catalog source, and the mockup overlay (model photo + print area) should all be config/DB-driven, not hardcoded. Then iPlusLove = uandiplus with CJ turned off + the size_inclusive filter on. Deleting a module, not building one.
